How Selections Work

In the past I have presented simplified descriptions of selections, hoping to provide an informative yet quick introduction to D3’s core concept—just enough to get started. This document takes a more comprehensive approach; rather than saying how to use selections, I will instead explain how selections are implemented. This may take longer to read, but it should dispel any magic and reveal everything you need to grasp selections completely.

#A Subclass of Array

You were probably told that selections are arrays of DOM elements. This is technically true but imprecise. To craft a more accurate definition, let’s start with a simple array of elements and incrementally add detail:

JavaScript doesn’t yet support array subclasses directly, so arrays are subclassed through prototype chain injection.

The first bit is that selections are a subclass of array, not plain arrays. This subclass provides methods to manipulate selected elements, such as setting attributes and styles.

Selections retain native array methods as well, such as array.forEach and array.map. However, you won’t often use native methods as D3 provides convenient alternatives, such as selection.each. A few native methods are overridden to adapt their behavior to selections (selection.filter and selection.sort).

#Groups of Elements

The other reason selections aren’t arrays of elements is that they are actually arrays of arrays of elements. A selection is an array of groups, and each group is an array of elements. For example, when you call d3.selectAll, it returns a selection with a single group, and this group contains the selected elements:

While the selection is a subclass of array, groups are plain arrays. They do, however, have a parentNode property which stores the shared ancestor of the group’s elements.

This is why you’ll sometimes see selection[0][0] to refer to the first node, although for reasons that will soon be apparent it’s better to use selection.node.

Selections returned by d3.select and d3.selectAll only have a single group, so the only way you to get a selection with multiple groups is selectAll followed by (one or more) selectAll. For example, selecting the rows of a table returns a selection with a single group: 

var tr = d3.selectAll("tr");

But if you then select the rows’ cells, you get a group for each row, grouping sibling cells: 

var td = tr.selectAll("td");
Although this selection’s groups all have four elements, in general a selection’s groups can have different numbers of elements. Some groups might even be empty!

Most of the time, you can ignore that selections are nested. Even when you pass a function to attr or style, the function is called for each element, so the only difference with a grouped selection is that the second argument to your function (typically i) is the within-group index rather than the within-selection index. The killer feature for nesting is the behavior of selection.data, which allows you to define data per-group, and more easily create hierarchical DOM elements from hierarchical data. We’ll revisit this topic when we get to data joins.

#Sparse Arrays

Sometimes it’s useful to derive a new selection from an existing selection while retaining the original grouped structure. An example of this is selection.select, which selects the first matching element within each selected element; select is also the basis of append and insert (see append.js and insert.js). If we amend the previous example to use select rather than selectAll, note that the result only has a single group: 

var td = tr.select("td");

To support this use case, selections can have null elements which are ignored for most operations:

Selections can contain null elements. This allows sparse selections which are missing elements. Most commonly this applies during a data-join, where the enter, update and exit selections are derived from an input selection and an array (or multiple arrays) of data. The enter and update selections match the indexes of the data; element i in the enter selection corresponds to datum i; if element i is in the update selection instead, element i in the enter selection is null.

#Elements and Data

Perhaps surprisingly, data is not a property of selections; data is a property of elements. This means that when you bind data a selection, the data is stored in the DOM rather than in the selection. In this way selections are transient: you can reselect elements from the DOM and they will retain whatever data was previously bound to them.

Note, however, that selection.select propagates data from parent to child, which can overwrite data that was previously bound.

Elements can have data bound to them. This is stored in __data__ and so isn’t strictly part of the selection; if you reselect elements from the document, they will retain whatever data was bound to them previously.

And of course, not every element may have data associated with it.

There are two forms of data-join, by index and by key. A data-join by index occurs when you call selection.data with only one argument; i.e., you specify the data but not the key function. A data-join by key occurs when you specify two arguments, the second being the key function.

Consider the more general join by key first. The key function is invoked for each element in the selection, and for each datum in the data. The key function returns a string key. If the key for an element matches the key for a datum, the element and the datum are considered “joined” and the element is put in the update selection.

The update selection is simply the return value of selection.data. Why not an explicit update method? Well, for brevity.

The enter selection contains placeholder nodes. What are these? Furthermore, the enter selection is not a full d3.selection, and only supports a subset of methods. Generally, an enter selection should only exist transiently; you are expected to call enter.append or enter.insert to immediately create a full selection.
